# Contributing to Documentation

Thank you for contributing to GenFeed documentation! This guide will help you understand what to document and how.

## What to Document

### ✅ User-Facing Apps Only

We document **only** user-facing apps that help users create content and improve their prompting:

- **Studio** - Content generation
- **Manager** - Content organization
- **Publisher** - Scheduling and publishing
- **Analytics** - Performance tracking
- **Automation** - Workflow automation
- **Dashboard** - Organization overview

### ❌ Do NOT Document

**Please do NOT create documentation for:**

- **Login App** - Authentication is automatic, no user docs needed
- **Admin App** - Internal administration tools, not for end users
- **Stock App** - Not launched yet, not publicly available

## Documentation Focus

Our documentation helps users:

1. **Use the apps effectively**
2. **Improve their prompting skills**
3. **Get better results from GenFeed**
4. **Understand features and workflows**

## Before Contributing

1. **Check Guidelines**: Read `.agent/SYSTEM/DOCUMENTATION-GUIDELINES.md`
2. **Verify Scope**: Ensure feature is user-facing and publicly available
3. **Focus on Users**: Write for end users, not developers or admins
4. **Improve Prompting**: Include prompting tips and best practices

## Documentation Structure

- **App Guides**: `docs/apps/` - Detailed guides for each app
- **Prompting Guides**: `docs/prompting/` - Prompting techniques and tips
- **Feature Guides**: `docs/features/` - Feature-specific documentation
- **Getting Started**: `docs/getting-started.md` - Onboarding guide

## Questions?

If you're unsure whether something should be documented:

1. Is it in Studio, Manager, Publisher, Analytics, Automation, or Dashboard?
2. Is it accessible to regular users (not admins)?
3. Is it publicly launched?

**If all yes → Document it**  
**If any no → Don't document it**
